# Working Workflow

This repository is meant to support a repeatable working loop.

## Cycle

1. `Frame the case`
   Name the system, purpose, boundary, and why it matters.
2. `Map the surface`
   Capture decisions, systems, data classes, stakeholders, and regulations.
3. `Design the controls`
   Define identity, authorization, approvals, fallback, logging, and evidence.
4. `Stress the design`
   Ask how it fails, who can override it, and what would be visible afterward.
5. `Turn it into proof`
   State the artefacts, logs, reviews, signatures, tickets, and runtime evidence expected.

## Deliverable quality bar

Each artefact should be:

- specific enough to argue with
- short enough to maintain
- tied to named owners
- connected to the real system
- explicit about gaps

## Common failure modes

- The use case is still marketing language instead of operational scope.
- Regulations are named, but not translated into design consequences.
- Human review is mentioned, but no role or decision boundary exists.
- Logging is mentioned, but no evidence consumer or retention need is defined.
- Security claims are broad, but machine identity or approval control is missing.
- The proof pack describes aspirations instead of inspectable artefacts.

## Healthy working pattern

The best rhythm is:

1. draft fast
2. challenge hard
3. tighten the artefact
4. move to the next file

That is exactly where AI assistance is useful if it is instructed well.
